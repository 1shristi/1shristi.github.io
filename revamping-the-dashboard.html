Revamping the Dashboard

for a client’s parking site management application

overview
Despite building an excellent demo application for site managers to track and manage vehicles across parking sites, the content and features were not clear enough for executive stakeholders and potential buyers to understand and navigate.

Assessing the usability of the application I gathered that the issue of poor user experience and navigation was due to a lack of affordances particularly on the main dashboard page.

Initial interviews focused on answering the the who-what-why of the product; taking any initial design constraints as an opportunity to guide the design process and timeline for creating the new interface.

Constraint 1: little qualitative and quantitative usability data Constraint 2: no time to arrange user interviews as new qualitative usability data
  • Decision: interviewed stakeholders regarding the current UI as two specific personas
    1) a product owner persona and
    2) as a new user persona

Constraint 3: design team of one, bootstrap design system, limited to ~4 weeks
  • Narrowed the scope to addressing the most impactful screens first, i.e. Overview dashboard layout
  • Had the opportunity to be a part of the team and adapt to the team dynamics in order to better advocate for incorporating design processes into their development cycle
solution
The bulk of the actual UX work (and thus the solution) involved analyzing and prioritizing the sidebar navigation menu items by how relevant they were for stakeholders and users.

With these affordances added, potential buyers had a clearer understanding of the application's value from the initial view (based on user feedback reported by the client).
Key UI decision: Gradual reveal
To avoid information overload, the user is introduced to information and data gradually
  • The map is the first item users see. It visually references the physical space that site managers work with (in their mental model).
  • Interacting with each pie chart reveals further location information within the map, enabling user-friendly association without confusion.
  • Pie charts frame the rest of the content displayed in the dashboard cards.
  • Each card summarizes information that encourages users to navigate to a seperate page to complete any associated tasks.
  • As a rule, previous read-only table data is shown in charts to reduce information overload.
Overview dashboard: selecting pie chart
  • One of the key features of the application is its ability to compare datasets; it was important to show this feature visually so as to guide the user through the vast amounts of information
  • Selecting a pie chart on the map introduces the user to further details about that specific location (including all its associated datasets across menu-items)
Overview dashboard: floor plans
  • To solve for secondary menues and wizard forms, I advocated for applying visuals that representing physical documents normally associated with the task, for example, physical drawings of parking sites
challenges information architecture
Challenge

Bespoke features were lost in the sheer amount of data tables across the 13 menu-items. And with little affordances to guide users, it was clear to see information architecture was one of the UX gaps. The following steps details my approach to prioritizing content for the purpose of designing a new dashboard.

Analysing the sidebar navigation

Menu items in the side bar were listed somewhat arbitrarily with each menu item related to a specific user task. Examining each screen provided insight into the kinds of tasks users completed.

For each screen linked from the Navigation sidebar

What can the user see?

What actions can the user perform (i.e. searching, exporting, etc.)?

Why is the information valuable for users?

What does the user want to do with the information?

Would the user need to perform any actions prior?

Interviewing stakeholders about the business

Defining the purpose of each screen helped contextualize the interviews with stakeholders and identify relationships between product, people, and the system. Interview answers clarified stakeholder assumptions regarding the target user.

challenges dataset relationships
Discovering complexities in the system

Of the narrowed down list of menu-items, some interactive components triggered changes to datasets in other areas of the system without the user's knowledge leading to confusion.

For example, if a user updated the parking fee attributes for a single vehicle, it may override other billing charges across accounts (unbeknownst to the user who made the change).

Defining relationships by the following tasks further reduced the number of menu-items to 4 (see below) and led to discovering ways to add affordances for the dashboard screen:

(1) the task the user completes in a menu-item, (2) the number of interactions associated to each menu-item, and (3) the impact the interaction has on other menu items

Space management and Site settings, Activity and Users

Relationship between menu-items

The simpler the relationship, the better the content will be for a dashboard.

Number of user actions to complete task vs respective priority of task

The higher the priority and the fewer interaction actions associated to it, the better the content will be for a dashboard.

Priority to complete task vs respective impact on system

The higher the priority and the less it triggers unknown changes across the system, the better the content will be for the dashboard.

To avoid complexities and confusion
  • Apply affordances to guide user around one dataset to another
  • Find opportunity to use spatial assets to reduce information overload
  • Use visuals and typography to organize different types of menu-item content
  • Where important information is far too complex to be in a dashboard, avoid showcasing more than necessary
Learnings from the project

The biggest challenge was the difficulty in mapping attributes and contingencies for dataset relationships by business value in a simpler way than how it was initially shown.

Besides feedback from product owners, defining the dataset by universal heuristic patterns (ex. more information leads to more confusion) was a quick win that ultimately guided the direction of the rest of the application.


Stakeholders had a general idea of the kind of information they wanted to see as part of the final design. I was hesitant to accept client's design requests without fully comprehending why (if at all) the changes would benefit the user.

In this case, I simply trusted the client's word because they even took the time to provide clear answers for their requests through extra meetings. This helped me clarify or update other parts of the design that the request impacted without delays to the overall timeline. It made for a pleasureable work experience with all the teams involved so I try to find (and reflect) this kind of empathetic way of working together.
